# 아키텍처 보고서

## 1. 목적

이 프로젝트는 한진 서버 원본 문서를 정리하고, 검색 가능한 구조화 데이터로 적재한 뒤, 프론트에서 근거 중심 검색을 제공하는 것을 목표로 한다.

핵심 원칙:
- LLM 없이 문서 ETL, 색인, 검색을 먼저 안정화한다.
- 원본 데이터와 가공 데이터, 실행 코드를 디렉토리 단위로 분리한다.
- 검색 결과는 문서 제목이 아니라 실제 근거 행과 원본 경로를 함께 반환한다.

## 3. 처리 파이프라인

1. `data/source/한진서버데이터`에서 원본 문서를 수집한다.
2. `code/scripts/preprocess_documents.ps1`가 파일을 `parse_ready`, `review`, `exclude`, `duplicate_skip`로 분류한다.
3. `code/scripts/rag_stage.rs`가 `parse_ready.csv`를 읽어 `data/rag_stage/files`와 `manifest.csv`를 만든다.
4. `code/evidence_backend`가 매니페스트를 DB에 반영한다.
5. 문서 파서가 문서 본문을 `documents`, `sections`, `tables_meta`, `table_rows`, `entities`, `document_profiles`, `row_search`에 적재한다.
6. `web_api.rs`가 SQLite 기반 검색 API를 제공한다.
7. 프론트는 행 단위 검색 결과와 문서 상세를 직접 조회한다.

## 4. 런타임 구성

### 전처리 계층

- 역할: 후보 문서 선별, 중복 감지, 제외 규칙 적용
- 주요 산출물:
- `data/preprocess_output/inventory_all.csv`
- `data/preprocess_output/excluded_auto.csv`
- `data/preprocess_output/review_queue.csv`
- `data/preprocess_output/parse_ready.csv`
- `data/preprocess_output/exact_duplicates.csv`

### 스테이징 계층

- 역할: 파싱 대상 문서를 안정적인 이름으로 복사하고 매니페스트 생성
- 특징:
- 문서 파일은 `doc_XXXXXX__원본이름.ext` 형식으로 보관
- 매니페스트에는 절대경로 기반 `rag_path` 기록

### 백엔드 계층

- 역할: 스키마 생성, 매니페스트 적재, 문서 파싱, 검색 API 제공
- 구현:
- `xlsx_ingest.rs`: `.xlsx`, `.xlsm`
- `ooxml_ingest.rs`: `.docx`, `.pptx`, `.pptm`
- `db.rs`: SQLite 스키마/조회/색인 동기화
- `web_api.rs`: 요약, 검색, 문서 상세, 행 상세 API

### 웹 계층

- `web_frontend`: 현재 웹 프론트 작업 공간
- `legacy_python_frontend`: 기존 Python/Streamlit 기반 프론트

## 5. 데이터 분리 원칙

- `code`: 실행 코드와 스크립트
- `web`: 사용자 UI
- `data/source`: 원본 데이터
- `data/preprocess_output`: 전처리 산출물
- `data/rag_stage`: 파싱 대상 스테이징 산출물
- `data/backend`: SQLite DB
- `data/quarantine`: 검색 대상에서 격리한 위험/불필요 파일

## 6. 현재 지원 포맷

- 전처리 허용: `.doc`, `.docx`, `.xls`, `.xlsx`, `.xlsm`, `.ppt`, `.pptx`, `.pptm`, `.pdf`, `.txt`
- 실제 파싱 지원: `.xlsx`, `.xlsm`, `.docx`, `.pptx`, `.pptm`

메모:
- `.pdf`, `.txt`, `.doc`, `.xls`, `.ppt`는 전처리상 후보가 될 수 있지만 현재 Rust 파서 적재 범위에는 아직 들어가지 않는다.

## 7. 성능/운영 특성

- 적재 워커 수는 `available_parallelism()` 기반으로 자동 결정된다.
- 8C 16T 환경에서는 논리 코어 전체를 활용하는 방향으로 동작한다.
- SQLite는 `WAL`, `busy_timeout`, 재시도 로직을 적용해 병렬 적재 충돌을 완화했다.
- 기본 경로는 상대경로 기준으로 동작한다.

## 8. 현재 상태 요약

- 검색 대상 적재 핵심 경로는 Rust 백엔드 중심으로 정리되었다.
- `pptm`은 실제 색인 대상에 편입되었다.
- 랜섬 흔적 파일은 `data/quarantine/ransom`으로 격리했다.
- 현재 실제 손상으로 인해 끝까지 파싱되지 못한 파일은 `4307`번 DOCX 1건이다.
