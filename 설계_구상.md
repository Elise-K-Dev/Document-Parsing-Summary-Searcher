# 설계 구상

## 1. 문제 정의

원본 데이터는 엑셀, 워드, 파워포인트, PDF, 이미지, 압축파일, 잠금파일, 바로가기, 중복본이 뒤섞여 있다. 그대로 검색에 넣으면 노이즈가 많고, 실제 업무 근거를 빠르게 찾기 어렵다.

따라서 이 시스템은 먼저 다음 질문에 답하도록 설계한다.

- 특정 작업이 실제로 수행되었는가
- 관련 문서는 어디에 있는가
- 어떤 행과 어떤 문구가 근거인가
- 비슷한 작업이나 인접 문서도 같이 찾을 수 있는가

## 2. 제품 방향

- 문서 전체를 막연히 검색하는 것이 아니라, 근거 행 중심으로 검색한다.
- 원본 파일을 직접 읽는 검색보다, 전처리와 구조화를 거친 후 DB에서 검색한다.
- 검색 결과는 항상 원본 경로와 함께 제공한다.
- 사람 검토가 필요한 파일과 자동 제외 파일을 분리한다.

## 3. 단계별 구상

### 1단계

- 파일 인벤토리 생성
- 자동 제외 규칙 적용
- 중복본 분리
- 파싱 가능 문서만 스테이징
- SQLite 기반 검색 MVP 구축

### 2단계

- 워드, 파워포인트 색인 확대
- 문서 상세 보기 강화
- 문서 검색과 행 검색을 분리
- 관련 문서 추천 품질 개선

### 3단계

- PDF, 구형 오피스 포맷 지원 여부 검토
- 프론트 UX 개선
- 운영 자동화 스크립트 정리

## 4. 사용자 관점 요구

- 검색창에 키워드를 넣으면 결과가 빨리 나와야 한다.
- 결과는 “왜 이 문서가 잡혔는지” 바로 이해 가능해야 한다.
- 문서 상세 화면에서 행, 셀, 주변 행, 원본 경로를 확인할 수 있어야 한다.
- 파일 정리 기준은 사람이 납득할 수 있어야 한다.

## 5. 설계 원칙

- 원본 보존: 원본은 `data/source`에 유지한다.
- 가공 분리: 전처리와 스테이징 산출물은 별도 폴더에 둔다.
- 추적 가능성: 각 문서는 `original_path`, `staged_path`, `sha256`로 추적한다.
- 보수적 자동화: 위험 파일은 삭제보다 제외·격리를 우선한다.
- 점진적 확장: 전처리 허용 포맷과 실제 파싱 지원 포맷을 분리 관리한다.

## 6. 현재 구상에서 실제 반영된 것

- `code / web / data / docs` 구조 분리
- 상대경로 기반 Rust 실행
- OOXML 워드/파워포인트 색인
- `pptm` 편입
- 병렬 적재
- 랜섬 흔적 격리

## 7. 아직 남은 구상

- PDF 파싱
- 구형 `.doc`, `.xls`, `.ppt` 처리
- 웹 프론트 기준 검색 UX 통합
- 문서/행/관련문서 랭킹 정교화

## 8. 결론

현재 구상은 “원본 문서 더미를 안전하게 정리한 뒤, 근거 중심 검색 시스템으로 바꾼다”에 맞춰져 있다. 즉 이 프로젝트의 본체는 단순 파일 뷰어가 아니라, 전처리와 색인 규칙이 명확한 문서 검색 시스템이다.
