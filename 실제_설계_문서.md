# 실제 설계 문서

## 1. 범위

이 문서는 현재 저장소에서 실제로 동작하는 구조를 기준으로 정리한 상세 설계 문서다. 계획 문서가 아니라 현재 구현 상태를 설명한다.

## 2. 디렉토리 설계

### 코드

- `code/evidence_backend`
- `code/scripts`

### 웹

- `web/web_frontend`
- `web/legacy_python_frontend`

### 데이터

- `data/source/한진서버데이터`
- `data/preprocess_output`
- `data/rag_stage`
- `data/backend`
- `data/quarantine`

### 보조 문서

- `docs`

## 3. 실행 경로 설계

Rust 백엔드는 `code/evidence_backend`에서 실행하는 것을 기본 가정으로 한다.

기본 상대경로:
- DB: `../../data/backend/evidence.db`
- 매니페스트: `../../data/rag_stage/manifests/manifest.csv`
- 스테이징 파일 루트: `../../data/rag_stage/files`

## 4. 전처리 설계

실행 파일:
- `code/scripts/preprocess_documents.ps1`

입력:
- `data/source/한진서버데이터`

출력:
- `inventory_all.csv`
- `excluded_auto.csv`
- `review_queue.csv`
- `parse_ready.csv`
- `exact_duplicates.csv`
- `preprocess_summary.md`

분류 규칙:
- `exclude`: 잠금파일, 임시파일, 랜섬 흔적, 바이너리/미디어, 미지원 확장자
- `review`: 확장자 없음, 복구본, 이름 키워드 검토 대상
- `parse_ready`: 파싱 대상
- `duplicate_skip`: SHA256 기준 완전 중복

현재 전처리 허용 확장자:
- `.doc`
- `.docx`
- `.xls`
- `.xlsx`
- `.xlsm`
- `.ppt`
- `.pptx`
- `.pptm`
- `.pdf`
- `.txt`

## 5. 스테이징 설계

실행 파일:
- `code/scripts/rag_stage.rs`

역할:
- `parse_ready.csv`를 읽어 실제 파싱 대상 파일만 복사
- `data/rag_stage/files/<family>/doc_xxxxxx__원본이름.ext` 형식으로 저장
- `manifest.csv`, `metadata.jsonl`, `failures.csv` 생성

중요 구현 포인트:
- 매니페스트의 `rag_path`는 절대경로로 기록한다.
- 그래서 백엔드 실행 위치와 무관하게 스테이징 파일을 찾을 수 있다.

## 7. 데이터 모델 설계

핵심 테이블:
- `documents`
- `sections`
- `tables_meta`
- `table_rows`
- `cells`
- `entities`
- `document_profiles`
- `row_search`

설계 의도:
- `documents`는 문서 메타데이터와 파싱 상태를 가진다.
- `sections`는 시트, 슬라이드, 문단 구획을 표현한다.
- `tables_meta`는 표 단위 구조 정보를 가진다.
- `table_rows`는 실제 검색의 중심 레코드다.
- `cells`는 셀 단위 세부 보기를 지원한다.
- `entities`는 날짜, 장비번호 같은 정규화된 검색 포인트를 담는다.
- `document_profiles`, `row_search`는 문서 검색과 행 검색 최적화에 사용된다.

## 8. 병렬 처리 설계

문서 적재는 `thread::available_parallelism()` 기준으로 워커 수를 자동 결정한다.

동작 방식:
- 문서 목록을 공용 큐처럼 소비
- 워커마다 별도 SQLite 커넥션 사용
- 문서 단위 트랜잭션 적용
- `database is locked` 발생 시 재시도

이 구조는 8C 16T 환경에서 가용 논리 코어를 자동 활용하는 것을 목표로 한다.

## 9. 포맷별 설계 차이

### 엑셀

- 시트 단위 섹션 생성
- 헤더 추정
- 행/셀 단위 적재

### 워드

- 문단 단위 적재
- 문서 전체를 하나의 섹션으로 관리

### 파워포인트

- 슬라이드별 섹션 생성
- 슬라이드 내 문단 텍스트를 행 단위 적재
- `.pptm`도 매크로 실행 없이 텍스트만 읽는다

## 10. 현재 알려진 제한

- HTTP 서버는 `TcpListener` 기반 단순 구현이며 요청을 순차 처리한다.
- PDF 계열은 전처리 후보에는 들어가지만 실제 적재는 아직 없다.
- 손상 원본 파일은 코드로 복구하지 않는다.
- 일부 오래된 문서 포맷은 아직 미지원이다.

## 11. 현재 운영 결론

지금 시스템은 “전처리 -> 스테이징 -> SQLite 적재 -> 근거 검색 API”의 뼈대가 실제로 굴러가는 상태다. 즉 개념 검증 단계는 지나서, 문서 포맷 확장과 웹 UX 개선이 다음 중심 과제가 되는 구조다.
